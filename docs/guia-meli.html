Seção 01
Visão Geral do Sistema

O NEX Inventory é um sistema web de gestão de inventário para a operação logística de NODOs (agências parceiras) da Mercado Livre. Roda diretamente no navegador — sem instalação, sem app store — e funciona em celular, tablet e computador.

🏪
Agências (NODOs)
Pontos de coleta parceiros. Fazem inventário físico de pacotes e sacas, expedem para o SVC e acompanham pendências.
🏭
Hub SVC
Centro de distribuição. Recebe de todas as agências, confirma volumes e acessa o painel Consulta.
👔
Time MELI
Administração central: cadastra NODOs, monitora operação via Consulta e resolve divergências de motoristas.

🎯 O que o NEX resolve

Antes do NEX, o controle de inventário era feito em planilhas e WhatsApp, sem rastreabilidade em tempo real e com alto risco de divergência entre agências e SVC.

Com o NEX, cada pacote é registrado digitalmente no momento da chegada ou expedição — com timestamp em BRT, operador e geolocalização. O painel Consulta permite cruzar dados de expedições vs recebimentos para identificar divergências de motoristas em tempo real.

🌍

URL de acesso

luishenrique785.github.io/NEX — Salve nos favoritos do celular. Funciona como PWA (pode ser "instalado" na tela inicial do smartphone).

Seção 03
Fluxo Operacional Completo
1

Agência recebe pacotes (Inventário Físico)

A agência escaneia cada pacote com câmera (@zxing/browser), digitação manual ou foto. O sistema bloqueia duplicatas e exibe pendências de dias anteriores. Timestamps salvos em BRT.

2

Agência registra movimentação de Sacas

Chegada ou expedição de sacas com quantidade, transportadora e placa. Histórico do dia exibido em BRT.

3

Agência expede pacotes para o SVC

Registra motorista (nome, CPF, placa, transportadora) + escaneia cada pacote. Modal de confirmação (não Alert.alert). Pacotes saem do inventário ativo da agência.

4

SVC recebe os pacotes

Operador SVC toca em "Receber Pacotes", escaneia todos os pacotes do lote. Modal de confirmação. Após confirmar, app volta automaticamente ao painel SVC.

5

Pacotes confirmados = pendências eliminadas

Ao confirmar recebimento no SVC, o status dos pacotes na agência muda de "Aguardando SVC" para "Recebido". A seção "Não Recebidos no SVC" da agência é atualizada.

6

MELI monitora via painel Consulta

Acessa o painel Consulta com senha. Analisa expedições, agrupa por agência, identifica divergências de motoristas e rastreia pacotes individuais. Exporta CSVs para análise.

✅

Rastreabilidade em tempo real

Cada operação salva: timestamp BRT, NODO de origem, código do pacote, dados do motorista. O painel Consulta cruza todas essas tabelas para mostrar o estado exato de cada pacote na operação.

📍

Seleção de NODO por geolocalização

Ao entrar como "Agência", o app solicita permissão de localização. NODOs com coordenadas cadastradas dentro de 1 km aparecem automaticamente no topo. Se nenhum for encontrado, busca manual está disponível. NODOs sem coordenadas precisam ser editados no painel admin (ver Seção 7).

📷

Scanner web com @zxing/browser

Na web, o scanner usa a biblioteca @zxing/browser (BrowserMultiFormatReader) que acessa a câmera via API do browser. Lê QR Code, Code128, EAN e outros formatos. Sistema de cooldown: mesmo código bloqueado por 2s, código diferente aceito imediatamente. O botão 🔄 troca câmera frontal/traseira.

Seção 05
Módulo SVC

O SVC é o hub central que recebe de todas as agências. Não seleciona NODO — opera globalmente. Após confirmar recebimento, o app volta automaticamente ao painel SVC (via router.replace('/svc')).

OperaçãoDados obrigatóriosResultado no banco
Receber pacotesCódigo do pacote (scan/manual/foto). Motorista/placa opcionaisRegistro em svc_recebimentos + svc_recebimentos_pacotes
Sacas — ChegadaQuantidade + Transportadora + PlacaRegistro em sacas_movimentos tipo "chegada"
Sacas — ExpediçãoQuantidade + Placa + TransportadoraRegistro em sacas_movimentos tipo "expedicao"
ConsultaSenha de acessoLeitura das tabelas para análise (somente leitura)
↩️

Navegação após confirmar

Ao confirmar o recebimento, o app usa router.replace('/svc') para voltar ao painel SVC imediatamente. Isso evita que o usuário fique preso na tela de recebimento após salvar.

Na tela do Consulta, um campo de senha é exibido antes de mostrar os dados. A senha está configurada em src/config.ts na variável CONSULTA_PIN. Apenas o time MELI e responsáveis pelo SVC devem ter acesso.

Interpretando divergências

Um motorista com "faltando" pode significar: (a) SVC ainda não escaneou (pacotes recentes em trânsito), (b) motorista devolveu à agência sem registro, (c) extravio real. Use esta aba para acionar o motorista ou a agência específica. Os IDs faltando são exatamente os que aparecem como "Não Recebidos no SVC" na tela de inventário das agências.

Seção 07
Gestão de NODOs

NODOs são as agências parceiras. Cada NODO tem um código único (ex: BRNMG838), nome, endereço e coordenadas geográficas para o filtro de proximidade. O painel admin é protegido por PIN de administrador.

🔒

Acesso restrito por PIN

A tela "Novos NODOS" na tela inicial pede PIN de administrador configurado em src/config.ts. Apenas o time MELI deve ter acesso a este painel.

Importar NODOs via CSV

1

Exportar CSV da planilha Google Sheets

Planilha Controle NODOS → Aba BASE - Nodos: Arquivo → Fazer download → CSV

2

Acessar painel admin

Tela inicial → ⚙️ NOVOS NODOS → Digite o PIN de administrador

3

Upload do CSV

Toque em "Selecionar CSV" → selecione o arquivo. O sistema parseia automaticamente as colunas corretas da planilha.

4

Geocodificação automática

Para cada NODO com endereço, o sistema consulta o Nominatim (OpenStreetMap) para obter lat/lng. Rate limit: 1 requisição/segundo. Um lote de 50 NODOs leva ~1 minuto.

📊

Formato do CSV da planilha

A planilha "Controle NODOS - BASE - Nodos" tem este formato de colunas:

  • Coluna A: ETA (ignorado)
  • Coluna B: Nó origem (ignorado)
  • Coluna C: Facility NEx = Código do NODO (ex: BRNMG838)
  • Coluna D: Modal Veículo (ignorado)
  • Coluna E: Nome Place = Nome do NODO
  • Coluna F: Endereço = Endereço completo para geocodificação
♻️

Idempotente — seguro para re-importar

O sistema verifica se o código do NODO já existe antes de inserir. Re-importar o mesmo CSV só adiciona os NODOs novos, sem duplicar os existentes.

Edição Manual de Coordenadas

NODOs que não foram geocodificados automaticamente (endereço ambíguo ou falha no Nominatim) aparecem com um badge vermelho "Sem coord." no painel admin. É possível editar as coordenadas manualmente clicando no botão 📍.

Seção 08
Banco de Dados — Estrutura

O banco está no Supabase (PostgreSQL). Todas as tabelas têm Row Level Security (RLS) habilitado com políticas de leitura/escrita via chave anônima. Timestamps salvos em UTC e exibidos em BRT (UTC-3) pela camada de apresentação.

TabelaDescriçãoColunas principais
nodos Cadastro de agências parceiras id, codigo, nome, endereco, cidade, estado, lat, lng, created_at
sacas_movimentos Movimentação de sacas (chegada e expedição) id, nodo_id, tipo ('chegada'/'expedicao'), quantidade, transportadora, placa, created_at
pacotes_inventario Inventário físico de pacotes por agência id, nodo_id, codigo_pacote, foto_url, created_at
pacotes_expedicoes Expedições de pacotes da agência para o SVC id, nodo_id, codigo_pacote, nome_motorista, cpf_motorista, placa, transportadora, expedicao_id, created_at
svc_recebimentos Sessões de recebimento no SVC (cabeçalho) id, nome_motorista, cpf_motorista, placa, transportadora, created_at
svc_recebimentos_pacotes Pacotes individuais de cada sessão de recebimento SVC id, recebimento_id, codigo_pacote, created_at

🔗 Como as tabelas se relacionam

pacotes_inventario → pacote inventariado na agência

pacotes_expedicoes → mesmo código ligado a um expedicao_id e motorista → saiu da agência

svc_recebimentos_pacotes → mesmo código escaneado no SVC → ciclo completo

O painel Consulta faz JOINs entre essas três tabelas para calcular status de cada pacote e divergências de motoristas.

📸

Storage de fotos

Fotos de pacotes ficam no Supabase Storage no bucket "package-photos". URLs públicas são salvas na coluna foto_url da tabela pacotes_inventario.

🔧

Como acessar o banco diretamente

Acesse app.supabase.com → Projeto gjjwcpdkdqvfgephwulk → Table Editor para ver e filtrar todos os registros. Use o SQL Editor para queries avançadas. Exemplo de query para ver pendências:

SELECT codigo_pacote FROM pacotes_expedicoes
WHERE codigo_pacote NOT IN (SELECT codigo_pacote FROM svc_recebimentos_pacotes)

Seção 09
Deploy e Infraestrutura

O sistema é implantado automaticamente via GitHub Actions a cada push na branch main. Zero configuração de servidor necessária.

1

Push na branch main

Qualquer commit na branch main dispara o workflow automaticamente.

2

GitHub Actions executa o build

Instala dependências e roda npx expo export --platform web. Gera a pasta dist/. O workflow também copia docs/ para dist/docs/ (guias HTML ficam disponíveis em produção).

3

Deploy automático para gh-pages

peaceiris/actions-gh-pages@v4 publica o conteúdo de dist/ na branch gh-pages.

4

Site disponível em ~2 minutos

GitHub Pages serve a branch gh-pages em luishenrique785.github.io/NEX. Usuários veem a nova versão ao atualizar a página.

⚠️

Configurações críticas do app.json

  • web.output: "single" — SPA mode, evita SSR que quebra no browser
  • experiments.baseUrl: "/NEX" — Necessário para assets corretos no GitHub Pages

🔑 Como fazer uma atualização

Para publicar qualquer mudança no sistema:

1. Edite o código no GitHub (UI web ou clone local)
2. Commit na branch main
3. Aguarde ~2 minutos
4. Verifique em luishenrique785.github.io/NEX

Não é necessário PC, npm ou qualquer ferramenta local — tudo roda via GitHub Actions na nuvem.

Segurança

  • As políticas de Row Level Security (RLS) garantem que dados só são acessíveis com a chave correta.
  • O PIN de admin e o PIN do Consulta são verificados client-side — são bloqueios operacionais, não de segurança crítica.
  • Para segurança real, considere adicionar autenticação Supabase Auth em versões futuras.
